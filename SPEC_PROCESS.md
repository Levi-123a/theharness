## 1. Brainstorming 关键节点

### 1.1 智能体追问了哪些好问题

brainstorming 技能通过逐个提问引导设计决策，以下是关键追问及其影响：

| # | 问题 | 我的回答 | 对设计的影响 |
|---|------|---------|-------------|
| 1 | 重点深入维度选哪个？ | 反馈闭环 | 确定了整个项目的核心贡献方向，所有后续设计围绕此展开 |
| 2 | 技术栈用什么？ | Python | 决定了 LLM SDK、测试框架、分发方式的选择 |
| 3 | LLM 供应商接哪个？ | 多供应商（默认 OpenAI） | 直接影响了 LLM 抽象层的设计——接口+实现分离 |
| 4 | 具体面向什么场景？ | 代码修复型 | 缩小了 scope，反馈闭环链路最短最清晰 |
| 5 | API Key 安全存储方案？ | 加密文件+主密码 | 确定了 CredentialManager 的实现方向 |
| 6 | 分发形态选哪种？ | Docker | 影响了 CI 配置和部署方案 |
| 7 | WebUI 形态？ | 终端流式+历史记录 | 确定了 FastAPI WebSocket 的架构 |
| 8 | 失败分类器区分哪些类型？ | 5 种（确认） | 直接定义了反馈闭环的核心逻辑 |
| 9 | 停机条件如何设计？ | 5 种（确认） | 确定了主循环的控制流 |
| 10 | 护栏拦截哪些操作？ | 5 类（确认） | 确定了 Guardrail 的正则规则 |
| 11 | 记忆存储什么？ | 4 种（确认） | 确定了 MemoryStore 的数据结构 |

**关键洞察**：brainstorming 技能的"一次一个问题"策略非常有效——它避免了信息过载，每个问题都建立在前一个回答的基础上，逐步收敛到一个清晰的设计。

### 1.2 哪些追问让我修正了原设想

- **LLM 供应商**：我原本倾向固定 OpenAI，但智能体提出"多供应商支持"选项并解释了"抽象层设计为可切换接口天然满足 mock 注入要求"——这让我意识到多供应商不只是功能扩展，更是 §A.4-A "可注入 mock 的 LLM 抽象层"的天然实现方式。采纳。
- **应用场景**：我原本想做"综合型"（支持多种场景），但智能体推荐"代码修复型"并引用 YAGNI 原则——这让我意识到 scope 聚焦比功能全面更重要，尤其是反馈闭环的深度需要场景聚焦来体现。采纳。
- **WebUI 形态**：智能体推荐"极简交互式"（YAGNI），但我选择了"终端流式+历史记录"——因为历史会话是用户故事 #5 的需求，不能省略。这里我推翻了智能体的推荐。

## 2. 至少 3 轮关键迭代的对话节选与处理决策

### 迭代 1：重点维度的选择

**智能体提议**：
> 建议作为重点的维度：治理（护栏/沙箱/HITL 状态机/范围围栏）、反馈闭环（确定性校验器+失败分类+多轮自我修正）、扩展（工具分发/多 agent 编排）。推荐反馈闭环。

**我的决策**：选择反馈闭环（B）。

**理由**：反馈闭环最契合 coding 场景的核心价值（运行测试→解析→自我修正），机制链路完整且可确定性测试。同时天然串联了"工具执行"和"治理"两个维度。这个选择直接决定了 SPEC §11 的领域与机制设计方向。

### 迭代 2：LLM 供应商与抽象层设计

**智能体提议**：
> 推荐多供应商支持（默认 OpenAI），抽象层设计为 LLMProvider 接口+具体实现，mock 是第二个实现。

**我的决策**：采纳（选项 3）。

**理由**：这个设计直接满足了 §A.4-A "可注入 mock 的 LLM 抽象层"的硬性要求。如果固定单一供应商，mock 注入会变成"打补丁"而非"接口设计"。多供应商支持让 mock 成为架构的一等公民，而非测试时的权宜之计。

**修正**：我原本想固定 OpenAI 以减少复杂度，但智能体的论证让我意识到"接口+实现分离"本身就是 mock 可注入性的保证。

### 迭代 3：SPEC 合规性检查——发现两处缺失

**用户要求**：
> 当前spec是否已完美符合task.md文档中要求

**我的处理**：逐项对照 TASK.md 检查后发现两处缺失：
1. §3.6 条件要求：含 WebUI 须说明所选设计系统 → 补充原生 HTML/JS 理由
2. §A.4-D 六个维度：须显式列出全部六个维度 → 补充 §11.3 六维度表

**决策**：立即补充。这暴露了一个重要教训——brainstorming 技能在引导设计决策方面很有效，但它不会主动检查设计是否覆盖了外部规范的所有要求。这需要人工对照 TASK.md 逐项检查。"设计确认"和"规范合规检查"是两个不同的步骤，不能合并。

## 4. 反思：brainstorming 技能的优缺点

### 做得好的地方

1. **一次一个问题**：避免了信息过载，每个问题建立在前一个回答上，逐步收敛。这比"一次性问完所有问题"有效得多——后者会让用户给出浅层回答，前者迫使深入思考每个决策。

2. **多选项+推荐+理由**：每个问题都提供 2-4 个选项，并给出推荐和理由。这让决策变得容易——即使我选择推翻推荐，推荐也提供了思考的锚点。

3. **分块呈现设计**：将设计分成 8 块（每块 200-300 字）逐步呈现，让我能在每块后审查和修正。这比一次性呈现完整设计更有效——后者会让人"看不完就批准"。

4. **YAGNI 原则贯穿**：brainstorming 不断提醒"不要做不需要的东西"，这帮助我避免了"综合型场景"这种 scope 蔓延。

### 让我不满的地方

1. **不会主动检查规范合规性**：brainstorming 技能引导设计决策，但不会主动对照外部规范（如 TASK.md）检查设计是否覆盖所有要求。这导致 SPEC 初稿遗漏了两处要求（§3.6 设计系统、§A.4-D 六维度）。需要人工补充"规范合规检查"步骤。

2. **设计块之间的连贯性**：分块呈现虽然便于审查，但块与块之间的逻辑连贯性有时不够清晰。例如，第 3 块（反馈闭环）和第 4 块（工具/护栏/记忆）之间的依赖关系需要读者自行推断。

3. **对"机制必须是代码"的强调不足**：brainstorming 在设计阶段没有足够强调 §A.4-B "机制是代码非提示词"的要求。这导致初始设计中对"哪些是代码机制、哪些可能滑向提示词"的边界不够清晰。需要在 SPEC 中显式补充（最终在 §11.5 补充了机制编码实现表）。

4. **缺乏对"冷启动验证"的预告**：brainstorming 阶段没有提前预告 §4.5 的冷启动验证要求。如果提前知道要用另一个 agent 测试 SPEC 的清晰度，可能会在 brainstorming 阶段更注重"可读性"而非"完整性"。

## 5. PLAN.md 生成过程

### writing-plans 技能的介入

brainstorming 完成后，触发 `writing-plans` 技能。该技能要求：
- 每个 step 是一个 2-5 分钟的动作
- 每个 task 包含：文件路径、TDD 步骤（写失败测试→验证失败→实现→验证通过→提交）
- 精确的文件路径和命令

### 关键决策

- 将 SPEC 分解为 14 个 task，每个 task 可由一个 subagent 在一次会话内完成
- 显式标注依赖图和可并行部分（Task 3-7,10 可并行；Task 12,13 可并行）
- 总计 77 个测试用例，全部使用 mock-LLM 驱动
- PLAN.md 持续更新协议：每完成一个 task 标记完成并附 commit hash

### 对 PLAN 质量的自评

- **优点**：task 颗粒度合适，依赖关系清晰，TDD 步骤明确
- **不足**：部分 task 的实现要点描述较精炼，可能需要 subagent 在执行时参考 SPEC 补充细节
- **预期风险**：Task 11（Agent Main Loop）依赖 8 个前置 task，是集成点，最可能暴露 spec 不清晰之处
